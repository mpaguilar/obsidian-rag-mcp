# Domain Glossary: Obsidian RAG

A comprehensive glossary of business and domain-specific terminology for the Obsidian RAG project.

---

## Core Entities

| Entity | Definition | Related Entities | Attributes |
|--------|------------|------------------|------------|
| **Vault** | A collection of Obsidian markdown documents; analogous to a notebook or workspace. Contains documents and serves as the root organizational unit. | Documents, Tasks | `name`, `description`, `container_path`, `host_path`, `created_at` |
| **Document** | An individual markdown file with content, metadata, and optional vector embedding for semantic search. The primary content unit. | Vault, Tasks | `file_path`, `file_name`, `content`, `content_vector`, `checksum_md5`, `kind`, `tags`, `frontmatter_json` |
| **Task** | A todo item extracted from markdown content, tracked with status, priority, and dates. Parsed from checkbox syntax in documents. | Document | `status`, `priority`, `due`, `scheduled`, `completion`, `description`, `tags`, `repeat` |
| **FrontMatter** | YAML metadata block at the beginning of markdown documents (between `---` delimiters). Contains structured metadata. | Document | `kind`, `tags`, plus arbitrary custom properties |
| **Tag** | Categorical labels extracted from content (prefixed with `#` in Obsidian). Used for organization and filtering. | Document, Task | String values, deduplicated, case-insensitive matching |

## Document Processing Terms

| Term | Definition | Usage Context |
|------|------------|---------------|
| **Ingestion** | The complete process of scanning vault files, parsing content, extracting metadata/tasks, generating embeddings, and storing in the database. | CLI, MCP Server, Services |
| **Scanning** | Recursive discovery of `.md` files within a vault path, excluding hidden files/directories. | Parsing/Scanner |
| **Parsing** | Extracting structured data from markdown: FrontMatter, tasks, tags, and metadata. | Parsing modules |
| **Checksum** | MD5 hash of document content used for change detection. Stored as 32-character hex string. | Change tracking |
| **Change Detection** | Comparing current file checksum with database to identify new, modified, or unchanged documents. | Ingestion flow |
| **Orphaned Document** | A document record in the database whose source file no longer exists in the filesystem. Cleaned up during ingestion. | Database maintenance |
| **Dry Run** | Simulation mode that processes files without writing to database. Used for testing and validation. | CLI, Ingestion |
| **Batch Processing** | Processing files in groups (default: 100) for memory efficiency and progress reporting. | Ingestion |
| **Max File Size** | 10MB limit for files; larger files are skipped during scanning. | Performance protection |
| **Soft Delete** | Not used - documents are hard-deleted when source files are removed (files are source of truth). | Data policy |

## Vector Search & Embeddings

| Term | Definition | Technical Details |
|------|------------|-------------------|
| **Embedding** | Vector representation of text content for semantic similarity search. Generated by LLM providers. | List of floats, default 1536 dimensions |
| **Vector Dimension** | The size of embedding vectors. Must match the embedding model's output dimension. | Default: 1536, Max: 2000 (pgvector limit) |
| **Semantic Search** | Search based on meaning similarity using vector embeddings and cosine distance. | Returns documents with similarity scores |
| **Cosine Distance** | Similarity metric for vector comparison (lower values = more similar). Used for ranking search results. | `1 - cosine_similarity` |
| **HNSW Index** | Hierarchical Navigable Small World index for fast approximate nearest neighbor search on vectors. | PostgreSQL pgvector extension |
| **pgvector** | PostgreSQL extension enabling vector data types and similarity search operations. | Required for production |
| **Similarity Score** | Numeric value (0-1) indicating how similar a document is to the query. Higher = more relevant. | Returned in search results |
| **Embedding Provider** | Service that generates embeddings (OpenAI, OpenRouter, HuggingFace). | Configurable per endpoint |

## Task Management Terms

| Term | Definition | Values / Format |
|------|------------|-----------------|
| **Task Status** | Current state of a task in the workflow. | `not_completed`, `completed`, `in_progress`, `cancelled` |
| **Task Priority** | Importance level of a task. | `highest`, `high`, `normal`, `low`, `lowest` (default: `normal`) |
| **Checkbox Status** | Visual indicator in markdown for task state. Maps to TaskStatus. | `[ ]` = not_completed, `[x]` = completed, `[/]` = in_progress, `[-]` = cancelled |
| **Due Date** | Date by which a task should be completed. | ISO date format (YYYY-MM-DD) |
| **Scheduled Date** | Date when a task is planned to be worked on. | ISO date format |
| **Completion Date** | Date when a task was marked as completed. | ISO date format |
| **Recurrence / Repeat** | Pattern for recurring tasks in rrule format (RFC 5545). | e.g., "FREQ=DAILY", "FREQ=WEEKLY;INTERVAL=2" |
| **Custom Metadata** | Additional key-value pairs extracted from task text using `[key:: value]` syntax. | Stored as JSONB |
| **Raw Text** | The complete original task line from the markdown document, including all metadata. | Preserved verbatim |
| **Description** | Clean task text with metadata and tags removed. | Human-readable task content |
| **Inline Tags** | Tags extracted from task text (e.g., `#important`). | Separate from document-level tags |

| Term | Definition | Usage |
|------|------------|-------|
| **Property Filter** | Filter condition on document FrontMatter properties using path notation. | `query_documents`, `get_documents_by_property` |
| **Tag Filter** | Filter for including/excluding documents by tags with match mode. | Document queries |

| **Has More** | Boolean indicating if additional results exist beyond current page. | Pagination response |
| **Next Offset** | Offset value for retrieving the next page of results, or `null` if no more pages. | Pagination response |

## Key Workflows

### 1. Document Ingestion Workflow
```
Scan Vault → Filter Valid Files → Parse FrontMatter → Extract Tasks 
→ Generate Embeddings → Compute Checksum → Store in Database 
→ Delete Orphans
```

### 2. Semantic Search Workflow
```
Receive Query → Generate Query Embedding → Vector Similarity Search 
→ Apply Filters (Property/Tag) → Return Ranked Results with Scores
```

### 3. Task Query Workflow
```
Receive Filter Criteria → Query Tasks Table → Join with Documents 
→ Apply Date/Status Filters → Return Paginated Results
```

### 4. Configuration Loading Workflow
```
Load Defaults → Load Config File → Interpolate Env Vars 
→ Apply CLI Flags → Validate Settings → Use Configuration
```

## Domain Invariants

1. **Vault Uniqueness**: Vault names must be unique (enforced at database level)
2. **Document Path Uniqueness**: `(vault_id, file_path)` pairs must be unique
3. **Foreign Key Constraints**: 
   - Documents cascade delete when vault is deleted
   - Tasks cascade delete when document is deleted
4. **Vector Dimension Constraint**: `vector_dimension ≤ 2000` (pgvector HNSW limit)
5. **Checksum Consistency**: Documents are only updated when MD5 checksum changes
6. **Task Status Mapping**: Checkbox characters map deterministically to TaskStatus enum
7. **Tag Normalization**: Tags are deduplicated, stripped of whitespace, and stored lowercase
8. **Pagination Limits**: `limit` parameter capped at 100 results per query
9. **Max Tasks Per Document**: 10,000 tasks maximum to prevent abuse
10. **File Size Limit**: Files > 10MB are skipped during scanning

| Term | Meanings | Context Disambiguation |

| **Tag** | 1) Document-level tags from FrontMatter<br>2) Task-level inline tags (e.g., `#important`)<br>3) Search/filter criteria | Check field context (document.tags vs task.tags) |
| **Path** | 1) `file_path` - relative to vault root<br>2) `container_path` - Docker internal path<br>3) `host_path` - external system path | Field name prefix clarifies usage |
| **Status** | 1) Task workflow status (not_completed, completed, etc.)<br>2) HTTP response status<br>3) Server health status | Context-dependent (TaskStatus enum vs HTTP) |
| **Filter** | 1) Property filter (FrontMatter matching)<br>2) Tag filter (include/exclude lists)<br>3) Query filter (combined parameters) | Model/parameter type indicates filter type |
| **Embedding** | 1) The vector representation (noun)<br>2) The process of generating vectors (verb)<br>3) The provider service | Context in code/comments clarifies |
